# テスト・アンチパターン

**このリファレンスを読むタイミング:** テストを書く・変更する、モックを追加する、本番コードにテスト専用メソッドを足したくなったとき。

## 概要

テストはモックの振る舞いではなく、本物の振る舞いを検証しなければならない。モックは分離するための手段であり、テストの対象ではない。

**核となる原則:** モックが何をするかではなく、コードが何をするかをテストする。

**厳格な TDD を守ることが、これらのアンチパターンを防ぐ。**

## 鉄の掟

```
1. モックの振る舞いをテストするな
2. 本番クラスにテスト専用メソッドを足すな
3. 依存関係を理解せずにモックするな
```

## アンチパターン1: モックの振る舞いをテストしている

**違反:**
```typescript
// ❌ BAD: モックが存在することをテストしている
test('サイドバーをレンダリングする', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**なぜダメか:**

- モックが動くことを検証しているだけで、コンポーネントが動くことは検証していない
- モックがあれば通り、モックがなければ落ちる
- 本物の振る舞いについて何も語らない

**human partner の指摘:** 「これはモックの振る舞いをテストしてない?」

**修正:**
```typescript
// ✅ GOOD: 本物のコンポーネントをテストする、またはモックを外す
test('サイドバーをレンダリングする', () => {
  render(<Page />);  // サイドバーはモックしない
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// あるいは、分離のためにサイドバーのモックが必要なら:
// モックそのものを assert しない - サイドバーが居る状態で Page の振る舞いをテストする
```

### ゲート関数

```
モックの要素について assert する前に:
  問う: 「これは本物のコンポーネントの振る舞いか、モックが存在しているだけか?」

  もしモックの存在をテストしているなら:
    停止 - assertion を消すか、コンポーネントのモックを外す

  本物の振る舞いをテストする
```

## アンチパターン2: 本番に混ざるテスト専用メソッド

**違反:**
```typescript
// ❌ BAD: destroy() はテスト用にしか使われていない
class Session {
  async destroy() {  // 本番 API のように見えてしまう!
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... cleanup
  }
}

// テストで
afterEach(() => session.destroy());
```

**なぜダメか:**

- 本番クラスがテスト専用コードで汚染されている
- 本番でうっかり呼ばれたら危険
- YAGNI と関心の分離に違反
- オブジェクトのライフサイクルとエンティティのライフサイクルを混同

**修正:**
```typescript
// ✅ GOOD: テスト用クリーンアップはテストユーティリティが扱う
// Session に destroy() を持たせない - 本番ではステートレス

// test-utils/
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// テストで
afterEach(() => cleanupSession(session));
```

### ゲート関数

```
本番クラスにメソッドを足す前に:
  問う: 「これはテストでしか使われない?」

  もしそうなら:
    停止 - 足さない
    代わりにテストユーティリティに置く

  問う: 「このクラスはこのリソースのライフサイクルを所有している?」

  もし違うなら:
    停止 - このメソッドは別のクラスにあるべき
```

## アンチパターン3: 理解せずにモックする

**違反:**
```typescript
// ❌ BAD: モックがテストのロジックを壊す
test('サーバの重複を検出する', () => {
  // モックが、テストが依存していた config 書き込みを止めてしまう!
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 例外を投げるはず - だが投げない!
});
```

**なぜダメか:**

- モックしたメソッドには、テストが依存していた副作用（config 書き込み）があった
- 「念のため」モックしすぎることで、本物の振る舞いが壊れる
- テストが間違った理由で通る、または不可解に落ちる

**修正:**
```typescript
// ✅ GOOD: 正しいレベルでモックする
test('サーバの重複を検出する', () => {
  // 遅い部分だけモックし、テストに必要な振る舞いは保つ
  vi.mock('MCPServerManager'); // 起動が遅いサーバの起動だけモック

  await addServer(config);  // config 書き込み
  await addServer(config);  // 重複検出 ✓
});
```

### ゲート関数

```
何かをモックする前に:
  停止 - まだモックしない

  1. 問う: 「本物のメソッドにはどんな副作用がある?」
  2. 問う: 「このテストはその副作用に依存している?」
  3. 問う: 「このテストが何を必要としているか、完全に理解している?」

  もし副作用に依存しているなら:
    より低いレベル（実際の遅い処理・外部呼び出し）でモックする
    または、必要な振る舞いを保つテストダブルを使う
    テストが依存している高レベルメソッドではない

  もしテストが何を必要としているか分からないなら:
    まず本物の実装でテストを実行する
    実際に何が起きる必要があるか観察する
    そのうえで、正しいレベルで最小限のモックを足す

  レッドフラグ:
    - 「念のためモックしておく」
    - 「これ遅そうだからモックしておく」
    - 依存関係を理解せずにモックする
```

## アンチパターン4: 不完全なモック

**違反:**
```typescript
// ❌ BAD: 部分モック - 自分が必要だと思ったフィールドだけ
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 抜けている: 下流で使われる metadata
};

// 後で: response.metadata.requestId にアクセスするコードで壊れる
```

**なぜダメか:**

- **部分モックは構造的な前提を隠蔽する** - 自分が知っているフィールドだけモックしている
- **下流のコードが含めなかったフィールドに依存しているかもしれない** - 静かに失敗する
- **テストは通るが統合で落ちる** - モックは不完全、本物 API は完全
- **誤った安心** - テストは本物の振る舞いについて何も証明しない

**鉄則:** 直近のテストが使うフィールドだけでなく、現実に存在する完全なデータ構造をモックする。

**修正:**
```typescript
// ✅ GOOD: 本物 API の完全性を再現する
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 本物 API が返すすべてのフィールド
};
```

### ゲート関数

```
モックレスポンスを作る前に:
  確認: 「本物 API のレスポンスはどんなフィールドを含んでいる?」

  すること:
    1. ドキュメント・サンプルから実際のレスポンスを調べる
    2. システムが下流で消費するすべてのフィールドを含める
    3. モックが本物のレスポンススキーマと完全に一致することを検証

  重要:
    モックを作るなら、構造全体を理解しなければならない
    部分モックは、抜けたフィールドにコードが依存したときに静かに失敗する

  不確実なら: ドキュメントに載っている全フィールドを含める
```

## アンチパターン5: 統合テストの後回し

**違反:**
```
✅ 実装完了
❌ テスト未着手
「テスト準備できました」
```

**なぜダメか:**

- テストは実装の一部。後回しの選択肢ではない
- TDD なら防げた
- テストなしに「完了」と言ってはいけない

**修正:**
```
TDD サイクル:
1. 失敗テストを書く
2. 通すように実装
3. リファクタ
4. その後で完了と宣言
```

## モックが複雑になり過ぎたとき

**警告サイン:**

- モックの setup がテストロジックより長い
- テストを通すために全部モックしている
- 本物のコンポーネントにあるメソッドがモックには無い
- モックが変わるとテストが落ちる

**human partner の問い:** 「ここでモックを使う必要がある?」

**検討:** 本物のコンポーネントを使った統合テストの方が、複雑なモックより単純であることが多い。

## TDD はこれらのアンチパターンを防ぐ

**なぜ TDD が効くか:**

1. **先にテストを書く** → 実際に何をテストしているか考えざるを得ない
2. **失敗を見届ける** → モックではなく本物の振る舞いを検証していると確認できる
3. **最小実装** → テスト専用メソッドが紛れ込まない
4. **本物の依存** → モックする前に、テストが本当に必要としているものが見える

**モックの振る舞いをテストしているなら、TDD に違反している** — 本物のコードに対して失敗を確認する前に、モックを足してしまっている。

## クイックリファレンス

| アンチパターン | 修正 |
|--------------|------|
| モックの要素を assert | 本物のコンポーネントをテストするか、モックを外す |
| 本番にテスト専用メソッド | テストユーティリティに移す |
| 理解せずにモック | まず依存関係を理解、モックは最小限 |
| 不完全なモック | 本物 API を完全に再現 |
| テストを後回し | TDD - テストが先 |
| 過剰に複雑なモック | 統合テストを検討 |

## レッドフラグ

- assertion が `*-mock` の testID を見ている
- テストファイルからしか呼ばれないメソッド
- モックの setup がテストの50%超を占める
- モックを外すとテストが落ちる
- なぜモックが必要か説明できない
- 「念のため」モックしている

## 結論

**モックは分離の道具であり、テストの対象ではない。**

TDD で「モックの振る舞いをテストしている」と気付いたなら、どこかで道を外している。

修正: 本物の振る舞いをテストするか、そもそもなぜモックしているのかを問い直す。

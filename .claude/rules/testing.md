# Testing Rules

## 基本方針

- AAA パターン（Arrange → Act → Assert）で書く
- 1テスト1アサート原則。複数の振る舞いをチェックしたい場合はテストを分ける
- テスト名は「何を、どんな状況で、どう振る舞うか」を日本語で記述する
  ```ts
  it("空文字の email を渡したとき、ValidationError を投げる", () => { ... });
  ```

## 配置と命名

- テストは対象ファイルと同じディレクトリに `<name>.test.ts` で配置
- 統合テストは `__tests__/` 配下に配置可
- フィクスチャは `__tests__/fixtures/` に分離

## モック

- モックは最後の手段。実コードで通せるなら通す
- モックの「振る舞い」をテストしない（実装をテストしているだけになる）
- 外部 API / DB / 時刻に依存する箇所のみモック化
- モックデータは実 API のレスポンス構造と一致させる

## 禁止事項

- production code にテスト専用のメソッド・分岐を入れない
- テスト間で状態を共有しない（各テストは独立して実行できること）
- `it.skip` / `it.only` をコミットしない
- スナップショットテストは UI コンポーネントのみで使用、ロジックには使わない

## 検証

- カバレッジは目標値ではなく「振る舞いが網羅されているか」で判断
- エッジケース（空・null・境界値・異常値）を必ず1ケース以上含める
